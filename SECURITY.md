# Security Policy

## Supported Versions

This project is maintained as a rolling, single-stream codebase.

| Version | Supported |
| --- | --- |
| `main` (latest) | Yes |
| Older commits/releases | No |

Security fixes are applied to current code on `main`.

## Reporting a Vulnerability

If you believe you have found a security vulnerability, please report it privately.

Preferred method:

1. Go to the repository Security tab.
2. Select "Report a vulnerability".
3. Submit a private report with clear reproduction steps and impact.

Please include, where possible:

- A clear description of the issue.
- Steps to reproduce.
- Affected file(s), script(s), or workflow(s).
- Expected impact and realistic exploit scenario.
- Any suggested remediation.

Do not open public issues for vulnerabilities that could expose users, credentials, infrastructure, or account access.

Use public issues only for non-sensitive hardening ideas, documentation improvements, or general security discussions that are safe to disclose.

## Scope and Handling

In scope (examples):

- Credential exposure or insecure secret handling.
- Authentication or authorisation weaknesses.
- Unsafe workflow or automation behaviour.
- High-impact dependency or supply-chain risk.

Out of scope (examples):

- Reports without actionable technical detail.
- Social engineering attempts.
- Non-security feature requests.

## Response Expectations (No SLA)

This repository is maintained on a best-effort basis and is not an enterprise service.

- There are no formal SLAs or guaranteed response times.
- Reports are reviewed as capacity allows.
- Best efforts will be made to respond within a reasonable timeframe.

Thank you for reporting issues responsibly and helping improve the project.
